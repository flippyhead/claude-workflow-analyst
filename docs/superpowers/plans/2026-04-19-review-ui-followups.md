## 2. Optional: re-scan / re-recommend buttons in the review UI

**Status:** Deferred. User said "Idk about the buttons" when proposing these.

**Why it might be worth adding:** Right now the review UI is read-only against the existing catalogue. To refresh, the user has to exit, run `/radar-scan` or `/radar-recommend` in Claude Code, then relaunch the UI. A button in the UI header could trigger a fresh scan/recommend without leaving the browser.

**Why it's non-trivial:**

- The scan and recommend workflows are *skills*, not standalone scripts. They involve WebFetch/WebSearch, catalogue enrichment logic, and LLM judgment calls (category assignment, scoring). There is no pure Node entry point that can be invoked from the server.
- Two options:
  1. **Dumb trigger + return to Claude Code:** Button writes a sentinel file (`~/.claude/radar/.refresh-requested`) and exits the server. The `radar-review` skill checks for the sentinel on exit and chains into `/radar-scan` or `/radar-recommend` before reporting the session summary.
  2. **Split the logic:** Extract the non-LLM parts of scan (fetching HN/GitHub/Anthropic feeds, dedup, enrichment heuristics) into the bundled workflow-analyzer CLI, and have the server shell out. This loses LLM-judged categorization and scoring — probably not worth it.

**Recommendation:** If we do this, go with option 1. Keep server + UI pure; let the skill orchestrate the refresh cycle.

**Open question before building:** is the "exit UI → rerun skill → user clicks URL again" loop acceptable, or do we want to keep the same browser tab open and have it auto-reload when the new catalogue lands? The second is more work (server needs to stay alive across scan runs, and the scan has to be invokable without the LLM in the loop).

## 4. Inline insight commentary on each card

**Why:** Right now the review UI shows `title`, `description`, `category`, `tags`, `score`, and the 4-part score breakdown — but not the *reasoning* behind the score. `/radar-recommend` already writes this reasoning to `catalogue.insights[]`:

```json
{
  "id": "insight-<ts>-<i>",
  "type": "recommendation",
  "observation": "what the data shows — cite the specific usage pattern or environment detail",
  "recommendation": "the concrete action to take",
  "evidence": ["score breakdown: goal=N, gap=N, recency=N, effort=N, total=N"],
  "relatedItems": ["<item id>"],
  "createdAt": "...",
  "status": "new"
}
```

The user sees a number but not the prose. The prose is the bit that makes "should I star this?" an actual decision instead of a guess from the title.

**What to change:**

- **Build an index** at load time in `index.html`: `insightsByItemId = {}` where each key is an item id and value is the `insights[]` entry whose `relatedItems[0]` matches. Latest wins on collision.
- **Render inline** on the card below the description, above the tag chips. Two short lines — `observation` and `recommendation`. Suppress entirely when the item has no matching insight (so dismissed / old items stay clean).
- **Styling:** light-gray left border, small italic type, `observation` prefixed "Why:" and `recommendation` prefixed "Try:". Keep it ≤ 2 lines each on a typical card width; let it wrap for long ones (don't truncate — the whole point is the reasoning).
- **Escape** both fields through `escapeHtml()` — insight strings come from the LLM and go into `innerHTML`.

**Non-goals:**

- Don't build a second tab (that's #3 above — separate, heavier work).
- Don't show `evidence` (redundant with the existing score breakdown `<details>`).
- Don't expose insight `status` or editing — insights are read-only artifacts of the recommend run.

**Open question:** if an item has multiple insights (multiple recommend runs over time), do we show only the latest, or a small stack? Latest-only is simpler and matches the freshness bias of `lastRecommended`.

## 5. Offload bulk per-item work to Haiku subagents

**Why:** The scan and recommend skills currently do a lot of formulaic per-item work in the main conversation, which runs on the session model (Opus by default for this user). Two pieces are Haiku-suitable:

- **Enrichment (Step 5 of `/radar-scan`):** For each new discovery, pick a `category` from a fixed vocabulary, generate 3-5 free-text `tags`, and write a 1-2 sentence `description`. No cross-item context needed. The fixed category vocab means Haiku can't hallucinate an invalid value if the system prompt lists them.
- **Scoring (Step 3 of `/radar-recommend`):** Four integer sub-scores per item against the user's context payload. The rubric is numeric and constrained (0-3 for goal, 0-3 for gap, 0-2 for recency, 0-2 for effort). The context payload is identical across items, so it caches well.

Both are per-item, bounded, and mechanical. The main-loop judgment calls (triage during scan: *is this release interesting or a patch bump?*; final ranking during recommend: *which of the 7+ scorers wins the "Act Now" slot?*) stay on the session model.

**What to change in `plugins/radar/skills/radar-scan/SKILL.md`:**

- Main loop does triage (keep Step 3's "is this release interesting or a patch bump?" judgment on the session model).
- For each candidate that survives triage, dispatch `Agent({ subagent_type: "general-purpose", model: "haiku", ... })` to run Step 5 enrichment. The subagent receives `{ title, url, source, releaseNotes? }` and returns `{ category, tags, description }`.
- Main loop assembles the final item object from enrichment output + the canonical schema fields (`id`, `status`, `discoveredAt`, etc.) and writes the catalogue.

**What to change in `plugins/radar/skills/radar-recommend/SKILL.md`:**

- Build one shared context payload (session-derived patterns, env, CLAUDE.md goals). Pass it to each scoring subagent.
- Step 3: dispatch one Haiku subagent per item with `{ item, context }` → returns `{ goalAlignment, usageGap, recency, effort, observation, recommendation }`. The two prose fields map directly into the insight schema used by §4 — no separate "write insight" pass needed.
- Step 4 (presentation / tier assignment / final ranking) and Step 5 (insight row assembly + catalogue write) stay in the main loop — that's where the judgment lives.

**Batching vs. one-at-a-time:**

- One-at-a-time: clean context, easy to parallelize with multi-tool-call, trivially debuggable. Costs more round-trips.
- Batched (e.g., 5-10 items per subagent call): fewer round-trips, but prompt assembly and response parsing get fiddly and one bad item poisons the batch.
- **Recommendation:** start one-at-a-time with `run_in_background: false` and parallel dispatch (multiple `Agent` calls in a single assistant turn). Revisit batching only if it measurably matters.

**Tradeoffs to flag:**

- Haiku is weaker at reading between the lines of release notes (e.g., figuring out a release is actually a rename of an existing project). Triage stays in the main loop specifically to avoid handing those judgment calls to Haiku.
- No cross-item memory inside subagents — each one sees only its item. Fine for scoring; means we can't do "this item supersedes that older one" reasoning inside a subagent. That logic, if we ever want it, belongs in the main loop's final pass.
- Cost model changes: currently one big prompt; after, many small prompts. Net should be cheaper (Haiku is ~1/5 the price of Opus) but only if subagent round-trips don't dominate.

**Open question:** should `/radar-scan` and `/radar-recommend` expose a `--no-subagents` flag for debugging, or do we trust the subagent path and drop the synchronous variant? Probably expose a flag — the skills are small and the fallback is useful when Haiku is unreachable.
